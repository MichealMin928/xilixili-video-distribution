# 抖音、小红书、快手、微信视频号聚合运营方案

更新日期：2026-07-18

## 一、需求结论

目标平台：抖音、小红书、快手、微信视频号。

需要分成两类工作：

1. **账号治理**：修改名称、头像、封面、简介，检查并删除或隐藏历史内容。
2. **日常运营**：一份素材生成四个平台的适配文案，统一排期、定时或批量发布，保存发布记录。

目前 GitHub 上已有项目能较好地解决第二类“日常运营”，但没有成熟方案能安全、稳定地跨四个平台完成第一类“账号治理”。账号治理仍应在各平台官方后台逐个执行；聚合工具主要负责内容、排期和发布。

## 二、GitHub 项目对比

| 项目 | 四平台覆盖 | 统一界面 | 定时/批量发布 | API/MCP | 修改账号简介 | 删除平台作品 | 主要风险 |
|---|---|---:|---:|---:|---:|---:|---|
| [AiToEarn](https://github.com/yikart/AiToEarn) | 是 | 是 | 是 | 是 | 否 | 否 | 账号授权和 Relay/API Key 的第三方信任；部分能力受平台接口限制 |
| [social-auto-upload](https://github.com/dreammis/social-auto-upload) | 是 | 旧版 Web UI；当前主线偏 CLI | 是 | 有限 | 否 | 否 | 非官方浏览器自动化，平台风控与页面改版会导致失效；旧版 Web UI 有安全问题 |
| [PostFlow](https://github.com/jefftko/PostFlow) | 是 | 有 | 是 | 未见成熟统一接口 | 否 | 未见完整支持 | 项目成熟度和维护规模相对较低，同样依赖浏览器自动化 |

### 1. AiToEarn

更接近真正的“多平台运营中台”：支持四个平台、统一内容管理、发布日历、批量生成、定时发布、账号连接，并提供 API/MCP。项目更新和使用规模都较好，适合作为本业务的首选聚合发布工具。

局限：它的重点是发布与排期，不是账号资料管理。其代码说明，删除本地发布记录并不等于删除平台上的作品。因此不能用它替代各平台官方后台做历史内容清理。

### 2. social-auto-upload

支持抖音、小红书、快手的视频和图文上传；微信视频号主要支持视频上传与定时发布。当前主线偏命令行，旧版 Web UI 已不是主要维护方向。

不建议部署或公开旧版 Web UI。该项目已有公开安全报告，涉及旧版后台无认证、Cookie 暴露、开放跨域等风险：[安全问题 #214](https://github.com/dreammis/social-auto-upload/issues/214)。此外还有无头登录二维码失效、平台页面改版和小红书风控等问题：[登录问题 #224](https://github.com/dreammis/social-auto-upload/issues/224)、[抖音改版问题 #230](https://github.com/dreammis/social-auto-upload/issues/230)、[小红书风控问题 #188](https://github.com/dreammis/social-auto-upload/issues/188)。

如以后将它作为备用通道，只应使用本机 CLI、保持有界面模式、Cookie 仅存本机、不得开放公网端口。

## 三、推荐架构

推荐采用以下分工：

1. **本项目文件夹作为内容总库和审计存档**
   - 保存选题、脚本、图片、视频、平台适配文案。
   - 保存每次资料修改、内容审核和发布结果。

2. **Codex 负责日常内容生产和运营编排**
   - 根据酒店投影业务生成图文、短视频脚本和封面。
   - 自动生成抖音、小红书、快手、视频号四套适配标题、正文、话题和发布时间。
   - 发布前检查敏感宣传口径，尤其是“携程指定、自营、合作”等需要证据支持的表述。

3. **自建“洗哩洗哩四平台运营台”作为发布中台**
   - 不再依赖 AiToEarn 或其他第三方聚合账号。
   - 四个平台在本机专用真实 Chrome 中分别登录一次，登录态仅保存在本机。
   - Codex 通过本项目 CLI/API 创建内容、预填四个平台、保存记录并在明确确认后执行发布。
   - 先使用预填模式验证页面适配，再逐个平台开放正式发布。

4. **平台官方后台负责账号治理**
   - 名称、头像、简介、主页封面等一次性修改。
   - 历史作品删除、隐藏或调整可见范围。
   - 账号认证、企业信息、私信和违规申诉。

## 四、安全与运营规则

1. 不向任何脚本或第三方提供账号密码；优先使用官方扫码或 OAuth 授权。
2. 首轮每个平台只发布一条测试内容，确认排版、清晰度、话题和定位正常。
3. 前 10 条内容保留人工确认后再发布；稳定后再启用批量排期。
4. 删除作品、修改账号资料、回复争议评论等高影响操作单独记录，不纳入无人值守批处理。
5. 登录凭据、Cookie、Token 不提交到 GitHub，不通过聊天或普通文档保存。
6. 本地服务只监听本机，不开放公网端口；定期撤销不用的账号授权。

## 五、建议实施顺序

1. 完成抖音、小红书、快手、视频号的账号定位统一和历史内容检查。
2. 启动并测试自建本地运营台，先连接一个平台进行预填验证。
3. 验证成功后依次接入其余三个平台。
4. 建立四平台内容模板、发布清单和变更记录。
5. 每周集中制作素材并排期，日常只处理审核、评论和线索。

## 六、最终选型

**最终选型：自建“洗哩洗哩四平台运营台”，由 Codex 直接编排真实 Chrome。**

不再安装或依赖第三方聚合发布项目。平台页面改版时，直接维护本项目内的四个平台适配器。

账号简介修改和历史内容清理不强行自动化，继续使用各平台官方后台完成，以降低误删、封控和凭据泄露风险。
